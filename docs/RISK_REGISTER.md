# Risk Register

| ID  | Risk                                                                                  | Likelihood | Impact   | Mitigation                                                                                                                                                                                      | Owner               |
| --- | ------------------------------------------------------------------------------------- | ---------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| R01 | Scope vs. deadline: full feature list is unbuildable in the remaining days            | High       | High     | Freeze to Must-Have tier (ADR-001); stretch features gated behind demo-path acceptance; day-by-day schedule in Execution Plan §10                                                               | Human owner         |
| R02 | Destructive edits: a tool action damages the analyzed repo                            | Medium     | Critical | Public reports are read-only; sample rehearsals require explicit confirmation and server-authoritative lifecycle validation; live changes remain branch/PR-gated (ADR-002)                      | Codex               |
| R03 | Analysis accuracy: garden misrepresents code health (false dead-code, wrong coverage) | Medium     | High     | Real tooling per signal, golden HealthReport snapshots, `analysis:validate` gate, conservative labeling ("possibly unused")                                                                     | Codex               |
| R04 | LLM-invented health: explanations assert things analysis didn't find                  | Medium     | High     | Explanations grounded in the HealthReport payload only; eval suite for Magnifying Glass outputs; invariant in AGENTS.md                                                                         | Codex               |
| R05 | API outage/cost during demo or judging                                                | Medium     | High     | Offline sample repo with pre-computed analysis + canned explanations; bounded reports, 30/IP/10-minute explanation throttling, timeout, and caching in Execution Plan §6                        | Codex               |
| R06 | Demo repo choice fails: license, size, or toolchain mismatch                          | Medium     | Medium   | Stage 0 selection criteria + fallback repo chosen up front                                                                                                                                      | Human owner         |
| R07 | Untrusted-code execution: analyzed repo runs hostile code during analysis/tests       | Low        | Critical | Static analysis only by default; any test execution sandboxed; no secrets in the analysis environment                                                                                           | Codex               |
| R08 | Incomplete submission: video/README/form miss a hard requirement                      | Medium     | Critical | `BUILD_WEEK_REQUIREMENTS.md` checklist; final verification pass scheduled a day before deadline                                                                                                 | Human owner         |
| R09 | Non-coder comprehension fails: the garden doesn't actually read as understandable     | Medium     | High     | Playtest with at least one non-coder (Execution Plan §13); Magnifying Glass copy reviewed for jargon                                                                                            | Human owner         |
| R10 | Before/after payoff underwhelms on video                                              | Medium     | Medium   | Choose a demo repo with visually dramatic fixable issues; rehearse the 90-second tending sequence; storyboard in §15                                                                            | Human owner         |
| R11 | Public analysis endpoint is abused or exhausts GitHub quota                           | Medium     | High     | Validate URLs, cap files/bytes, apply best-effort per-IP throttling, cache reports for five minutes, time-limit GitHub requests, and retain the offline fallback                                | Codex               |
| R12 | Demo-fork change evidence is delayed by analyzer confidence or external GitHub state  | Medium     | High     | Keep demo-rehearsal mode explicit; never claim a PR or heal optimistically; calibrate absolute/dynamic entrypoint detection before selecting a real change; carry the gate into submission risk | Codex               |
| R13 | Learning questions accidentally become trivia or block non-coder progress             | Medium     | High     | Keep questions authored, short, report-grounded, and three-tiered; provide hints/retries with no fail state; run a non-coder read-through before acceptance                                     | Human owner + Codex |
| R14 | World controls distract from the evidence-based garden map                            | Medium     | Medium   | Keep movement deterministic and cosmetic; tool stations are named affordances, while plants, roots, and health remain pure HealthReport projection                                              | Codex               |
| R15 | Anonymous command replay or cross-user command collision                              | Medium     | High     | Server-issued opaque command tokens, server-side state registry, proof binding, expiry, and replay regression tests                                                                             | Codex               |
| R16 | Explanation cache returns content for the wrong report                                | Medium     | High     | Canonical validated report-content cache identity and unknown-node validation before cache lookup                                                                                               | Codex               |
| R17 | Large public reports render overlapping plants                                        | Medium     | Medium   | Deterministic occupied-slot resolution with regression coverage for colliding node IDs                                                                                                          | Codex               |
| R18 | Teaching lessons drift from the app or are too difficult for the intended age band    | Medium     | High     | Pin a reviewed teaching-repository commit, keep an offline fixture, validate lesson metadata, and require Grade 1–12 human review                                                               | Human owner + Codex |
| R19 | Classroom fixture accidentally exposes secrets, student data, or unlicensed content   | Low        | Critical | Keep lessons synthetic and public, scan before publication, forbid credentials/student data, and record asset provenance                                                                        | Human owner + Codex |
| R20 | Dense controls or evidence obscure the game world and confuse the interaction model   | Medium     | Medium   | Keep global controls in the top toolbar, require approach plus E/Enter, collapse detailed evidence, and run desktop/mobile playtests                                                            | Codex               |
| R21 | Python fixture content drifts from the question bank or proposed output              | Medium     | High     | Validate one authored 20-issue fixture against question IDs, difficulty counts, answer choices, excerpts, fixes, and integrity snapshots before every release | Codex |
| R22 | Static map paths and collision disagree or trap a target                            | Medium     | High     | Use an explicit collision grid, 2–3 tile paths, deterministic reachability tests, and desktop/mobile human walk tests | Codex |
| R23 | Minimal question dialogue still overwhelms beginner learners                         | Medium     | High     | Enforce the exact dialogue element contract, remove legacy labels, use short copy, and require Grades 1–5 human comprehension testing | Human owner + Codex |
| R24 | Safe output controls imply that the fixture or repository was changed                | Low        | Critical | Keep the fixture read-only, make every copy/download/diff action explicit, and test that no repository write occurs | Codex |
