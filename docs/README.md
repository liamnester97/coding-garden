# Workspace Documentation Map

There is one execution roadmap. The outer `docs/` directory contains that canonical roadmap and
this map; active supporting records live inside the nested application repository.

| Material | Use | Authority |
| --- | --- | --- |
| `EXECUTION_PLAN.md` | Long-form product blueprint and submission roadmap | Canonical |
| `_source/.../repo/docs/BUILD_WEEK_REQUIREMENTS.md` | Submission checklist | Active |
| `_source/.../repo/docs/GARDENER_JOURNEY.md` | Demo narrative and user journey | Active |
| `_source/.../repo/docs/RISK_REGISTER.md` | Risk register | Active |
| `_source/.../repo/docs/adr/` | Architecture decision records | Active |
| `_source/.../repo/PLAN.md` and `STATUS.md` | Current stage and project snapshot | Trackers |

The old source-package copy is preserved as
`_source/Code_Garden_Source_Package_v1/Code_Garden_Codex_Package_v1/docs/FROZEN_BASELINE.md`.
It is historical input, not an execution document. There is no second active execution plan.

When reference material and the active implementation records differ, preserve the product intent
from the blueprint but follow the nested repo's current plan and record any material change in its
decision log.
